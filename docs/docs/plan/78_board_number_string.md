# Issue #78 掲示場番号の文字列化 実行計画

## 全体像

- **目的**: 掲示場番号フィールドを文字列型に変更し、`01-2` のようなブロック＋掲示場番号形式を正しく扱えるようにする。
- **背景**: 現状は `boards.board_number`（および関連ドメイン）が整数型前提で、フロント/バック双方のバリデーションが整数チェックのみなため、現場で使われている `xx-x` 形式を保存できない。
- **スコープ**:
  - PrismaスキーマとPostgreSQLカラム（boards / board_import_rows / ステージングテーブル）の型変更、および既存データのキャスト。
  - ドメイン/DTO/リポジトリ/差分アルゴリズム/CSVパーサー/各種UI（地図・テーブル・レビュー）の `boardNumber` を `string | null` として扱う修正。
  - CSV/手動入力に対するバリデーション仕様のアップデートと重複判定ロジックの更新。
  - 関連ドキュメント（ER図・スキーマ表・インポート手順）の更新。

## 発見と驚き

- `boards.board_number` は `NOT NULL` な整数型だが、アプリ側ドメインは `number | null` を許容しており、将来的な未設定値サポートを見越している。文字列化と併せて `NULL` も正式に扱えるようにする必要がある。
- CSVインポートでは `boardNumber` をキーに重複・既存マッチングを行っているため、単純な型変換だけでなく、ハイフン付き文字列でも安定して比較できる正規化ロジックが必須。
- 既存のステージングSQL（`boards_import_stage` など）やドキュメントにも整数型前提の記述が散見されるため、実装と同時にドキュメント更新が必要。

## 決定ログ（YYYY-MM-DD HH:MM JST）

- 2025-11-08 21:42 JST: 掲示場番号フォーマットは「数字のみ」または「数字 + ハイフン + 数字」を基本とし、前後の空白を除去した上で比較・保存する方針で進める。
